# NewLife.Office 竞品分析报告

## 1. 概述

本报告对比 NewLife.Office 与市面上主流 .NET 办公自动化库，覆盖 **Excel、Word、PPT、PDF、Markdown、RTF、ODS、Email、iCalendar、vCard、EPUB、XPS** 以及 OLE2/BIFF8 老式二进制格式，从功能覆盖、性能、依赖、许可等维度进行综合评估，为 NewLife.Office 的功能规划提供参考。

## 5. Excel 竞品优劣势分析

### 5.1 EPPlus

**优势**：功能最全面的开源方案，API 设计友好，文档丰富。  
**劣势**：v5 起商业使用需购买许可证；不支持 xls 格式；内存占用较高。

### 5.2 NPOI

**优势**：开源免费，支持 xls/xlsx，功能覆盖广泛，社区活跃。  
**劣势**：包体积大（~10MB），依赖多，API 较底层，学习曲线陡峭，内存占用高。

### 5.3 ClosedXML

**优势**：API 友好、代码可读性好，MIT 许可。  
**劣势**：依赖 Open XML SDK（较重），不支持 net45，不支持 xls。

### 5.4 MiniExcel

**优势**：极致轻量，流式读写，内存占用极低，支持模板填充和对象映射。  
**劣势**：样式支持有限，不支持合并单元格、冻结窗格等高级特性。

### 5.5 Aspose.Cells

**优势**：功能最全，企业级品质，支持几乎所有 Excel 特性。  
**劣势**：**价格高昂**（单开发者License数千美元），闭源，包体积大。

### 5.6 API 代码易用性对比

以下对比各库完成最典型 Excel 操作所需代码量与风格，直观体现 NewLife.Office 的开发效率优势。

#### 对象集合导出

```csharp
// ✅ NewLife.Office（ExcelHelper 最简洁模式）— 1 行完成
ExcelHelper.SaveAs("report.xlsx", users);
```

```csharp
// ✅ NewLife.Office（ExcelWriter 完整控制）— 样式随行设置
using var writer = new ExcelWriter("report.xlsx");
writer.WriteObjects("Sheet1", users, new CellStyle { Bold = true, Background = "4472C4", ForeColor = "FFFFFF" });
writer.Save();
```

```csharp
// EPPlus — LoadFromCollection 完成映射，样式需逐步设置
using var package = new ExcelPackage("report.xlsx");
var sheet = package.Workbook.Worksheets.Add("Sheet1");
sheet.Cells["A1"].LoadFromCollection(users, PrintHeaders: true);
package.Save();
```

```csharp
// NPOI — 无内置对象映射，需完整手写反射循环（约 20 行）
var workbook = new XSSFWorkbook();
var sheet = workbook.CreateSheet("Sheet1");
var props = typeof(User).GetProperties();
var header = sheet.CreateRow(0);
for (var i = 0; i < props.Length; i++) header.CreateCell(i).SetCellValue(props[i].Name);
var rowIdx = 1;
foreach (var u in users)
{
    var row = sheet.CreateRow(rowIdx++);
    for (var i = 0; i < props.Length; i++)
        row.CreateCell(i).SetCellValue(props[i].GetValue(u)?.ToString());
}
using var fs = File.Create("report.xlsx");
workbook.Write(fs);
```

```csharp
// MiniExcel — 最简洁，但不支持任何样式设置
await MiniExcel.SaveAsAsync("report.xlsx", users);
```

#### 对象集合导入

```csharp
// ✅ NewLife.Office（ExcelHelper 最简洁模式）— 1 行完成
var list = ExcelHelper.Query<User>("report.xlsx").ToList();
```

```csharp
// ✅ NewLife.Office（ExcelReader 显式控制）— 自动按列名/DisplayName/Description 映射
using var reader = new ExcelReader("report.xlsx");
var list2 = reader.ReadObjects<User>().ToList();
```

```csharp
// EPPlus — 需指定列映射关系，代码量中等
using var package = new ExcelPackage("report.xlsx");
var sheet = package.Workbook.Worksheets[0];
var list = sheet.Cells["A1:Z1000"].ToCollectionWithMappings(
    row => new User { Name = row.GetValue<String>(1), Age = row.GetValue<Int32>(2) },
    options => options.HeaderRow = 0);
```

```csharp
// MiniExcel — 同样简洁
var list = await MiniExcel.QueryAsync<User>("report.xlsx");

// NPOI — 无内置支持，需手写列名→属性映射（约 20-30 行）
```

#### 模板填充

```csharp
// ✅ NewLife.Office（ExcelHelper 最简洁模式）— 1 行完成
ExcelHelper.SaveByTemplate("output.xlsx", "template.xlsx", new { Name = "张三", Date = DateTime.Today, Total = 9800m });
```

```csharp
// ✅ NewLife.Office（ExcelTemplate 显式控制）
var tpl = new ExcelTemplate("template.xlsx");
tpl.Fill("output.xlsx", new Dictionary<String, Object>
{
    ["Name"] = "张三", ["Date"] = DateTime.Today, ["Total"] = 9800m
});
```

```csharp
// MiniExcel（模式接近，语法略有差异）
var data = new { Name = "张三", Date = DateTime.Today, Total = 9800m };
await MiniExcel.SaveAsByTemplateAsync("output.xlsx", "template.xlsx", data);

// EPPlus / NPOI 无原生模板填充能力，需自己实现占位符替换逻辑
```

#### 流式读取大文件

```csharp
// ✅ NewLife.Office — IEnumerable 逐行 yield，内存极低
using var reader = new ExcelReader("bigfile.xlsx");
foreach (var row in reader.ReadRows())   // row: Object?[]
    Process(row);
```

```csharp
// MiniExcel — 同样支持流式，API 风格相似
await foreach (var row in MiniExcel.QueryAsync("bigfile.xlsx"))
    Process(row);

// EPPlus / ClosedXML — 全量加载到内存，不适合超大文件
```

#### 单元格样式设置

```csharp
// ✅ NewLife.Office — 值对象风格，一次构建跨行复用
var style = new CellStyle
{
    Bold = true, FontColor = "FF0000", Background = "FFFF00",
    HorizontalAlignment = HorizontalAlignment.Center,
    Border = CellBorderStyle.Thin
};
writer.WriteRow(null, new Object[] { "总计", 9800m }, style);
```

```csharp
// EPPlus — 每个属性单独设置，代码量多但可精细控制
var cell = sheet.Cells["A1"];
cell.Style.Font.Bold = true;
cell.Style.Font.Color.SetColor(Color.Red);
cell.Style.Fill.PatternType = ExcelFillStyle.Solid;
cell.Style.Fill.BackgroundColor.SetColor(Color.Yellow);
cell.Style.HorizontalAlignment = ExcelHorizontalAlignment.Center;
cell.Style.Border.Bottom.Style = ExcelBorderStyle.Thin;
```

```csharp
// NPOI — 需在 workbook 级别预先创建样式对象，API 源自 Java 风格
var font = workbook.CreateFont();
font.IsBold = true;
font.Color = IndexedColors.Red.Index;
var cellStyle = workbook.CreateCellStyle();
cellStyle.SetFont(font);
cellStyle.FillForegroundColor = IndexedColors.Yellow.Index;
cellStyle.FillPattern = FillPattern.SolidForeground;
cellStyle.Alignment = HorizontalAlignment.Center;
cell.CellStyle = cellStyle;
```

> **小结**：NewLife.Office 提供 ExcelHelper 静态入口（单行即可完成导入/导出/模板，媂美 MiniExcel 最简洁用法）；同时内置完整样式、图表、高级特性支持，远超 MiniExcel；以値对象风格的 API 比 EPPlus/NPOI 减少 60–80% 代码量。

### 6.3 Word 竞品优劣势分析

#### 6.3.1 Open XML SDK

**优势**：微软官方出品，MIT 许可，完整支持 OOXML 规范，类型安全，长期维护。  
**劣势**：API 极为底层和冗长，开发效率低，学习曲线陡峭，任何简单操作都需要大量代码。  
**亮点**：对 docx 底层结构有完全控制力，适合需要精确操作 XML 的场景。

#### 6.3.2 NPOI

**优势**：同时支持 doc（97-2003）和 docx 格式，Apache 2.0 免费，功能覆盖面广。  
**劣势**：包体积大（~10MB），依赖多，API 移植自 Java POI 风格不够 .NET 原生，内存占用高。  
**亮点**：唯一同时支持 doc 和 docx 的免费开源库。

#### 6.3.3 MiniWord

**优势**：极致轻量，API 简洁，专注模板填充场景，Apache 2.0 许可。  
**劣势**：功能非常有限，不支持读取，不支持页面设置、页眉页脚等，仅适合简单模板场景。  
**亮点**：与 MiniExcel 风格一致，几行代码完成模板填充，支持图片和表格标签。

#### 6.3.4 OfficeIMO

**优势**：MIT 许可，基于 Open XML SDK 的高层封装，API 比原生 SDK 友好得多。  
**劣势**：项目相对年轻，社区活跃度低，部分功能不够成熟，依赖 Open XML SDK。  
**亮点**：在 Open XML SDK 之上提供了更接近业务的 API 抽象。

#### 6.3.5 Aspose.Words

**优势**：功能最全的 Word 操作库，支持所有格式互转（docx→pdf/html/image），内置 Mail Merge 引擎。  
**劣势**：商业许可价格高昂（开发者License数千美元），闭源，包体积大（~40MB）。

### 8.3 PDF 竞品优劣势分析

#### 8.3.1 iText 7

**优势**：.NET 生态中功能最全面的 PDF 库，支持创建、编辑、表单、签名、PDF/A 等企业级特性，文档丰富。  
**劣势**：**AGPL 许可**要求使用者也开源，否则需购买商业许可（价格不低）；API 较复杂。  
**亮点**：表单填充、数字签名、PDF/A 合规等企业级功能是其核心竞争力。

#### 8.3.2 QuestPDF

**优势**：现代化 Fluent API 设计，开发体验极佳，MIT 许可（年收入<$1M），支持热重载预览，社区活跃度极高（12k+ Stars）。  
**劣势**：**仅支持创建**，不能读取或编辑已有 PDF；年收入>$1M 的公司需商业许可。  
**亮点**：C# 声明式布局、自动分页、组件复用的设计理念，在 PDF 生成领域代表了最先进的开发体验。

#### 8.3.3 PdfSharp / MigraDoc

**优势**：MIT 许可，轻量，PdfSharp 提供底层绘图 API，MigraDoc 提供文档模型（段落/表格/图片），可搭配使用。  
**劣势**：PdfSharp 无高层表格 API（需手动绘制线条），文本提取能力有限，无表单/签名支持。  
**亮点**：PDF 合并功能简洁高效；MigraDoc 可同时输出 PDF 和 RTF。

#### 8.3.4 PdfPig

**优势**：Apache 2.0 许可，专注 PDF 文本提取，支持逐字/逐行提取并保留位置信息，适合数据挖掘场景。  
**劣势**：**只读**，不能创建或编辑 PDF。  
**亮点**：文本提取的精度和位置信息获取能力在免费库中最优。

#### 8.3.5 Aspose.PDF

**优势**：功能最全，支持创建/编辑/转换/表单/签名/OCR 等全部 PDF 操作，零原生依赖。  
**劣势**：商业许可价格高昂，包体积极大（~50MB），闭源。  
**亮点**：HTML→PDF 转换、PDF→Word/Excel 反向转换的能力是其独特卖点。

## 11. 结论

NewLife.Office 定位为 **功能全面的 .NET 办公自动化库**，致力于用 MIT 开源协议为社区提供覆盖 Excel、Word、PPT、PDF 四大格式的高质量办公自动化能力。

### Excel 领域

当前版本已实现样式、合并、冻结、筛选、对象映射、DataTable、模板填充、图片、页面设置等核心能力。后续将补齐公式、图表等主流功能。

### Word 领域

市面上缺少一个**功能全面且免费**的 Word 操作库：
- MiniWord 功能过于简单（仅模板填充）
- NPOI/Open XML SDK 太重或太底层
- Aspose.Words 价格高昂

NewLife.Office 将以直接操作 ZIP + XML 的方式实现全面的 docx 读写能力。

### PPT 领域

.NET 生态中 PPT 操作库选择极为有限：

- Open XML SDK 太底层，开发效率低
- ShapeCrawler 功能还在完善中
- NPOI 的 PPT 模块成熟度不足
- Aspose.Slides 价格高昂

NewLife.Office 将填补这一空白，提供易用且功能全面的 PPT 操作能力。

### PDF 领域

PDF 库的许可证问题是行业痛点：

- iText 7 的 AGPL 限制了闭源项目使用
- QuestPDF 对大企业收费，且仅支持生成
- PdfSharp 功能有限，PdfPig 仅读取

NewLife.Office 以 MIT 许可提供全面的 PDF 操作能力，填补市场空白。

### OLE2/BIFF8 领域

实现老式二进制格式的 .NET 库较少且质量参差：

- **NPOI**（Apache 2.0）：最完整的 xls/doc/ppt 支持，但体积大（~10MB），API 底层（来自 Java 移植），内存高；
- **ExcelDataReader**（MIT）：专注 xls 只读，轻量（<1MB），不支持写入；
- **POIFS/POI 生态**：Java 系，.NET 中无原生对等实现。

NewLife.Office 将从零实现 OLE2/CFB 容器层作为共用基础，并在其上分别实现 BIFF8（.xls）、MS-DOC（.doc）、MS-PPT（.ppt）解析，确保与 NPOI 同等兼容性，并在零依赖和 MIT 许可方面保持优势。

## 13. 综合定位与竞争优势

### 10.1 各格式领域总结

当前版本已实现样式、合并、冻结、筛选、对象映射、DataTable、模板填充、图片、页面设置等核心能力。后续将补齐公式、图表等主流功能。

### Word 领域

市面上缺少一个**功能全面且免费**的 Word 操作库：
- MiniWord 功能过于简单（仅模板填充）
- NPOI/Open XML SDK 太重或太底层
- Aspose.Words 价格高昂

NewLife.Office 将以直接操作 ZIP + XML 的方式实现全面的 docx 读写能力。

### PPT 领域

.NET 生态中 PPT 操作库选择极为有限：

- Open XML SDK 太底层，开发效率低
- ShapeCrawler 功能还在完善中
- NPOI 的 PPT 模块成熟度不足
- Aspose.Slides 价格高昂

NewLife.Office 将填补这一空白，提供易用且功能全面的 PPT 操作能力。

### PDF 领域

PDF 库的许可证问题是行业痛点：

- iText 7 的 AGPL 限制了闭源项目使用
- QuestPDF 对大企业收费，且仅支持生成
- PdfSharp 功能有限，PdfPig 仅读取

NewLife.Office 以 MIT 许可提供全面的 PDF 操作能力，填补市场空白。

### 扩展格式领域

传统办公自动化库聚焦于 Excel/Word/PPT/PDF，扩展格式（Markdown/RTF/ODS/Email/iCalendar/vCard/EPUB/XPS）的高质量免费实现在 .NET 生态中较为分散或缺失。NewLife.Office 通过统一的零依赖方式覆盖这些格式，在 B 端系统集成场景（邮件处理、日历互通、联系人导入导出、电子书生成）形成独特竞争优势。

### 10.2 综合定位

NewLife.Office 作为 **.NET 平台最全面的办公自动化库**，统一提供 Excel + Word + PPT + PDF + Markdown + RTF + ODS + Email + iCalendar + vCard + EPUB + XPS + OLE2/BIFF8（.xls/.doc/.ppt）的操作能力，以**零外部依赖、全框架支持（net45+）、MIT 免费许可**为核心优势，成为 .NET 平台最具竞争力的开源办公自动化方案。
