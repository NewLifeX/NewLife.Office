## 10. NewLife.Office 差异化定位

### 10.1 核心竞争力

| 维度 | NewLife.Office 优势 |
|------|-------------------|
| **功能全面** | Excel + Word + PPT + PDF 四大办公格式统一在一个库中 |
| **零外部依赖** | 不依赖 Open XML SDK / PDFium 等第三方库，直接操作文件结构 |
| **全框架** | net45 ~ net10 全覆盖，竞品大多放弃 net45 |
| **免费** | MIT 许可，无商业风险（对比 EPPlus/iText/Aspose） |
| **统一 API** | 四大格式 API 风格一致，学习成本低 |
| **高性能** | 直接操作底层结构，避免中间层开销 |
| **智能类型** | 内置长数字保护、身份证保护、百分比识别（Excel） |
| **生态集成** | 与 NewLife.Core/XCode 等深度集成 |

### 10.2 目标赛道

NewLife.Office 定位为：

> **功能全面、高性能、零门槛的 .NET 办公自动化库**

- **Excel**：覆盖主流 Excel 操作功能，支持样式、公式、图表、数据透视表等
- **Word**：覆盖主流文档创建与编辑功能，支持排版、表格、图片、目录等
- **PPT**：覆盖主流演示文稿操作功能，支持幻灯片创建、形状、表格、图片等
- **PDF**：覆盖主流 PDF 操作功能，支持创建、读取、编辑、合并拆分等

让开发者以 MIT 许可、零外部依赖完成绝大多数办公自动化任务，不再为商业许可困扰。

### 10.3 竞品对标

#### Excel

- **vs EPPlus**：完全免费，无商业许可限制，框架兼容性更好
- **vs NPOI**：API 更简洁现代，学习成本更低
- **vs ClosedXML**：无 Open XML SDK 依赖，支持 net45
- **vs MiniExcel**：功能远超（样式、图表、公式等全面支持）

#### Word

- **vs MiniWord**：功能远超（读取、页面设置、目录、页眉页脚等全面支持）
- **vs NPOI**：API 更简洁现代，不引入 Java 风格
- **vs Open XML SDK**：高层封装，开发效率高
- **vs Aspose.Words**：完全免费，MIT 许可

#### PPT

- **vs ShapeCrawler**：无 Open XML SDK 依赖，功能更全面
- **vs NPOI**：API 更简洁，PPT 模块更完善
- **vs Open XML SDK**：高层封装，开发效率数倍提升
- **vs Aspose.Slides**：完全免费，MIT 许可

#### PDF

- **vs PdfSharp**：功能更全面（文本提取、合并拆分、水印等）
- **vs iText 7**：完全免费，无 AGPL 限制
- **vs QuestPDF**：支持读取和编辑（QuestPDF 仅生成），支持 net45
- **vs PdfPig**：同时支持读取和创建（PdfPig 仅读取）

## 11. 结论

NewLife.Office 定位为 **功能全面的 .NET 办公自动化库**，致力于用 MIT 开源协议为社区提供覆盖 Excel、Word、PPT、PDF 四大格式的高质量办公自动化能力。

### Excel 领域

当前版本已实现样式、合并、冻结、筛选、对象映射、DataTable、模板填充、图片、页面设置等核心能力。后续将补齐公式、图表等主流功能。

### Word 领域

市面上缺少一个**功能全面且免费**的 Word 操作库：
- MiniWord 功能过于简单（仅模板填充）
- NPOI/Open XML SDK 太重或太底层
- Aspose.Words 价格高昂

NewLife.Office 将以直接操作 ZIP + XML 的方式实现全面的 docx 读写能力。

### PPT 领域

.NET 生态中 PPT 操作库选择极为有限：

- Open XML SDK 太底层，开发效率低
- ShapeCrawler 功能还在完善中
- NPOI 的 PPT 模块成熟度不足
- Aspose.Slides 价格高昂

NewLife.Office 将填补这一空白，提供易用且功能全面的 PPT 操作能力。

### PDF 领域

PDF 库的许可证问题是行业痛点：

- iText 7 的 AGPL 限制了闭源项目使用
- QuestPDF 对大企业收费，且仅支持生成
- PdfSharp 功能有限，PdfPig 仅读取

NewLife.Office 以 MIT 许可提供全面的 PDF 操作能力，填补市场空白。

### OLE2/BIFF8 领域

实现老式二进制格式的 .NET 库较少且质量参差：

- **NPOI**（Apache 2.0）：最完整的 xls/doc/ppt 支持，但体积大（~10MB），API 底层（来自 Java 移植），内存高；
- **ExcelDataReader**（MIT）：专注 xls 只读，轻量（<1MB），不支持写入；
- **POIFS/POI 生态**：Java 系，.NET 中无原生对等实现。

NewLife.Office 将从零实现 OLE2/CFB 容器层作为共用基础，并在其上分别实现 BIFF8（.xls）、MS-DOC（.doc）、MS-PPT（.ppt）解析，确保与 NPOI 同等兼容性，并在零依赖和 MIT 许可方面保持优势。
